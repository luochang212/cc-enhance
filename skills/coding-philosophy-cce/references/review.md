# 审查哲学（熵减）

代码审查的核心张力：**不是所有问题都值得修**。看到一个东西可以改，不等于应该改。每次修改都有成本——引入 bug 的风险、破坏一致性的可能、让代码变复杂而非变简单的危险。

## 核心原则

### 收益必须大于副作用

对每个潜在修复问三个问题：

1. **减少代码量同时保持可读性吗？** 提取一个两处使用的三行表达式值得。提取两个会分别演化的组件不值得。
2. **修复真实的正确性或命名问题吗？** 未使用的 import、误导性命名、死代码——无论代码量多少都修。
3. **抽象增加的复杂度比消除的多吗？** 一个带 6 个 prop 和条件分支的共享组件，省了 4 行 JSX，是净亏损。

## 判断启发式

### 显式 > 短

一组相似函数共享平凡结构（调用 API → 使缓存失效），不值得用工厂函数包装。每个函数五六行、独立可读；工厂函数会增加类型参数和参数接线，模糊每个变体的实际行为。**可见的重复好于不可见的意图。**

### 语义重复 ≠ 机械重复

两段看起来一样的代码不一定是重复：

- **不同层的相同类型**——反序列化类型和公开返回类型字段完全一样，但它们属于不同契约。合并会让公开 API 和上游变更耦合。这是**隔离**，不是浪费。
- **相同值但不同含义**——两个 key 映射到相同值可能编码了不同状态。合并它们会丢失「这个情况被考虑过且有意识映射」的显式信号。**语义清晰 > 消除一行。**

### 不要为假想的未来设计

没有辅助函数、抽象层、配置开关——除非**现在就有第二个调用方**。「以后可能需要」不是有效理由。**三行相似代码 > 一个过早的抽象。**

### 修复规模和问题规模匹配

- **两行的 bug 不值得重构整个模块。** 修完就走。
- **一个经过 8 次提交、长出 6 只手的 50 行函数**——这才值得重构。理解它的成本已经超过了清理它的成本。
- 测试标准：*如果只修问题本身，下一个人还能理解吗？* 能 → 补丁。不能 → 周围代码也是问题的一部分，重构是合理的。

### 遵循已有模式，要么全改

一个有问题的模式要么是项目级惯例，要么是项目级问题。修一个实例制造不一致；不修维持现状。如果这个模式没有造成实际 bug，别碰。如果造成了，单独一个 commit 全改——不要和功能开发混在一起。

## 审查标准

### 永远值得修

- lint/fmt/typecheck 错误——零争议
- 未使用的 import——死代码
- 死代码分支——相同分支的三元表达式、不可达代码
- 误导性命名
- 无意义包装——只转发给另一个函数的 `useCallback`，和另一个函数完全相同签名的函数体

### 值得评估（收益 > 副作用？）

- 重复逻辑——相同的非平凡表达式出现在 2+ 处
- 重复的 JSX 结构
- 近乎相同的函数——只有布尔值差异
- 跨文件的重复样板

### 通常不值得修

- 会分别演化的两处重复
- 简单 HTML 外壳的重复——带 class 的 `<div>` 不是有意义的重复
- 单次使用的抽象机会
- 有语义差异的机械重复——不同层职责的两个相同字段 struct、相同值不同含义的 key
- 项目级模式的半修复——修一个留一个是在制造不一致
- 不减少重复的「提取到文件」——把只在一处用的代码移到单独文件是文件重排，不是重构

## 审查流程

### 结构

- **Summary**：审查的高层概述
- **Critical**：bug、安全问题、破坏性变更
- **Improvements**：更好的代码质量或性能的建议
- **Nitpicks**：格式或风格问题（可选）
- **Conclusion**：明确的 Approve / Request Changes

### 并行读，顺序写

审查时并行读所有文件——多个 agent 同时看不同文件收集发现。修复时顺序写——两个 agent 同时编辑同一文件会静默覆盖。**并行发现，顺序修复。**

### 语调

建设性、专业、友好。解释**为什么**要求修改。批准时，明确肯定贡献的价值。
