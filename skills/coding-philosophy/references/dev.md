# 开发哲学（熵增）

写新代码的核心张力：**每行代码都是负债**。它今天解决问题，明天就要被维护、理解、调试。加得越多，系统越重。

## 原则

### 用最少代码解决问题

不要加没人要的功能。不要为单次使用建抽象。不要为不可能发生的场景写错误处理。不要添加没人要求的「可配置性」。每一层额外的封装、依赖、间接层都必须挣回自己的成本——**用更少代码表达更多是本事**。

### 只碰必须碰的

不改相邻代码。不修不是你搞坏的东西。匹配已有风格，即使你更想用另一种写法。只删你的改动导致不再需要的 import。

### 收益必须大于副作用

做任何选择前，问：**收益是否超过副作用？** 写代码不是比谁加得多，是比谁用最少的复杂度达到目的。

### 从第一性原理出发

把问题剥到可验证的事实，再向上推理。「大家都这么做」不是事实。

### 写之前先想

陈述假设。展示权衡。如果存在多种理解，全列出来——不要默默选一个。如果不确定，停下来问。改动之前问：这会引入 bug 吗？这比需要的更复杂吗？收益超过副作用吗？

### 一致性是承重结构

基于不同现实假设的层各自通过检查，但合在一起会失败。设计决策改变时，找到每个假设旧方案的地方——**旧代码是还没暴露的 bug**。

### 识别局部最优

当一系列增量改动暴露出更彻底的解决方案时，停下来和用户讨论再动手。不要默默把一个小任务变成大规模重构。

### 验证完再说完

不要说运行过如果没运行。如果无法验证，直说。

### 测试跟着代码走

在同一轮改动中写好测试或更新已有测试。不要攒到以后。「不重要所以不用测」的代码才是最容易出 bug 的。

## 做 vs 不做

| 做 | 不做 |
|------|------|
| 从第一性原理推理 | 盲目跟随惯例 |
| 陈述假设和权衡 | 默默选择一种理解 |
| 三行相似代码放原地 | 为一个用途建抽象 |
| 匹配已有代码风格 | 沿途「顺手」重构 |
| 一致地修改全局模式 | 只修一个实例制造不一致 |
| 验证完再声称完成 | 假装运行过 |
